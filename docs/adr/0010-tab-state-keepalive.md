# Tab state preservation via in-app DOM keep-alive

When a Space tab was activated, the URL was hardcoded to `/space/$spaceId` (the Space home), so switching from Tab A to Tab B and back lost Tab A's deep route. The deeper complaint that motivated this: once AI writing ships, a tab switch would terminate any in-flight AI stream. We preserve the full **Tab state** (see `CONTEXT.md`) by keeping every visited route subtree mounted in the DOM, hidden via CSS when inactive.

The chosen implementation is **hand-rolled in-app keep-alive**: a single TanStack Router with `useRouter()` + `<RouterContextProvider>` snapshotting per cached route, all cached routes rendered simultaneously, CSS `hidden` toggling between them. `KeepAliveProvider` lives at `__root.tsx` (above both `TitleBar` and `_space.tsx`) so that `useTabState()` is available to the title bar for `getLastRoute` / `clearSpaceTabState`. The cache Map is **shared across all Spaces**; eviction is by path-prefix (`/space/<id>`) when a Space tab closes. Cache key is the full route pathname. Nothing is evicted until a Space's tab is closed.

**Scope of preservation**: every visited route within a Space stays alive — `useState`, form drafts, scroll position, focus, modal state, future AI Chat instances. Closing the tab destroys that Space's cache; closing the app destroys everything (only the active Space's home route is restored on restart, via the existing `__root.tsx:38-45` startup logic — unchanged). The URL bar follows the active tab's last route: clicking Tab A navigates back to Tab A's last path (tracked in a `useRef<Map<SpaceId, string>>` at `__root.tsx`), not `/space/$spaceId`.

## Considered options (rejected)

- **Multi-webview (one OS webview per Space)** — Tauri v2 supports `Window::add_child(WebviewBuilder)` behind the `unstable` Cargo feature flag (still `unstable` in 2.11.x as of mid-2026, with core repaint fixes still landing in June 2026). Carries ~50–100 MB memory tax per Space, amplifies the documented WebView2/webkitgtk `eval`/IPC leaks across N webviews, and **has no z-order API** — `langwatch/kanban-code` (the one real-world precedent doing exactly per-tab OS webviews) comments verbatim: *"there is no z-order API in Tauri 2.x, so chrome must render around the child rect, never overlapping it."* This single missing primitive kills the browser-tab UX. Rejected.
- **Nested `<RouterProvider>` with `createMemoryHistory` per Space** — TanStack Router is architected around a single router instance per app. Discussion #2103 demonstrates the failure mode: URL updates and loaders fire, but the inner router's matches never commit to the DOM. The predecessor library's maintainer calls multi-router composition *"an anti-pattern, and straight-up not supported for good reason."* Rejected as architecturally unsupported.
- **Community library `tanstack-router-keepalive`** — implements the same clone-context approach we hand-roll, but ships with Issue #3 (`Maximum call stack size exceeded` from `lodash.clonedeep` on the router's `processedTree.segmentTree` circular refs) and known `useSearch()` staleness in cached routes (the staleness is fundamental to context cloning, not a library bug). Our needs (per-Space cache scope, close-tab eviction by prefix) are off-label for its API. Rejected.
- **`react-activity-keepalive-kit`** — built on React 19.2's experimental `<Activity>` primitive (formerly Offscreen). Betting on an unstable React API. Rejected for now; revisitable when `<Activity>` stabilises.
- **Persistent Tab state across app restart** — React component state has no general serialisation path (`useState` may hold `FormData`, `AbortController`, DOM refs, closures). Cost/value is poor for a novel-writing app where drafts auto-save to DB and the user's mental model is "close app = stop work". Rejected; restart restores the active Space's home only.

## Consequences

- **`useSearch()` returns stale values inside cached/hidden routes.** Audited: zero `useSearch` usage in `src/` today, so the caveat is currently moot. If URL search params are introduced under `/space/$spaceId/*` later, they must subscribe via `useRouterState()` directly, not via `useSearch()`.
- **Close-tab cleanup contract**: keep-alive cache for the closed Space is destroyed immediately; React Query cache is left untouched (TanStack Query GC handles it; reopen uses stale-then-refetch); pending mutations run to completion (do not cancel — would interrupt DB transactions); no "unsaved changes" warning (matches today's behaviour; that's a separate feature). A `onSpaceClosed(spaceId)` hook is reserved on the keep-alive registry for the future AI Chat registry to call `chat.stop()` + `chats.delete(spaceId)`.
- **No memory safety valve.** The cache grows unbounded within a Space's lifetime. Worst-case math (5 Spaces × 20 routes × ~10 MB ≈ 1 GB) leaves headroom under WebView2's ~2 GB process ceiling. If hit, WebView2 crashes and `__root.tsx`'s existing startup logic restores the active Space's home on next launch. A dev-only `console.warn` fires when cached subtree count exceeds 50; oxlint's `no-console: warn` rule keeps it out of production builds.
- **`title-bar.tsx` `handleActivate` / `handleClose` change shape.** Both currently call `navigate({ to: "/space/$spaceId", params: { spaceId } })`; after this ADR they navigate to `getLastRoute(spaceId) ?? /space/$spaceId`. `handleClose` additionally calls `clearCacheForSpace(closedId)` after `closeSpace.mutateAsync`.
- **Cache eviction uses a nonce, not Map identity.** `KeepAliveProvider` stores its cache in a `useRef<Map>` (mutations don't trigger re-renders — necessary because `setCachedNode` fires during `KeepAliveOutlet`'s own render). Eviction (`clearByPrefix`) mutates the Map in place, so without a signal, `KeepAliveOutlet` would never reconcile after eviction unless an unrelated re-render (e.g. `SpaceLayout`'s `useSession()` subscription) happened to pass through. A `useReducer`-backed `evictionNonce` is bumped on every successful `clearByPrefix` and threaded through context; `KeepAliveOutlet` subscribes to it, making eviction reliable regardless of host-layout render behaviour.
- **`display: contents` on the visible cache wrapper.** The active cached subtree's wrapper `<div>` uses `display: contents` (hidden ones use `display: none`). `contents` makes the wrapper invisible to layout — its children become flex/grid items of the grandparent (`_space.tsx`'s `relative flex flex-1` container), preserving `flex-1` / `h-full` on cached route roots. This works today because the grandparent has `position: relative`, so `position: absolute` descendants still find a containing block. **Invariant to preserve**: if a future cached route introduces `position: sticky` or relies on the wrapper for `overflow` clipping, `display: contents` will not establish a scroll container — revisit the wrapper strategy at that point (alternatives: `display: flex; flex: 1; min-height: 0` for visible, or per-route wrapper opt-out).
- **Per-pathname caching duplicates layout routes.** Because the cache key is the full pathname, pathless layout routes like `_world.tsx` (which renders `WorldSidebar`) are included in every cached entry under `/space/$id/world/$wid/*`. Observable consequence: `WorldSidebar` internal state (expanded groups, search filter text, scroll position) is **per-route**, not shared across sibling world routes. A user who collapses the "Characters" group on `/world/xyz/characters` will see it expanded again on `/world/xyz/locations`. This is the designed trade-off of Q2 ("every visited route stays alive") and the simplest correct implementation; nested keep-alive (cache at the `_world.tsx` level) would share sidebar state but is significantly more complex and not pursued.
