# 章节设计文档：第11章 扩展系统 —— 不改源码给 Agent 加能力

> **目标读者**：看过前 10 章的开发者，理解 Agent Loop / 工具 / 消息 / 事件驱动，但**没有插件系统设计经验**——"工厂函数"、"throwing stubs"、"事件总线"这些插件系统常见模式对他们来说是新的
> **目标**：读完之后，读者能：① 写一个最简扩展；② 理解 Pi 为什么用"事件总线 + 两阶段绑定"这套设计；③ 能在自己的项目里借鉴这套插件架构思路

## 1. 本章一句话

读者看完后脑子里的认知变化：

> **扩展系统不是"魔法注入"，而是 Pi 在工作流的 8 个关键点 emit 事件——把"改变行为"的能力通过事件总线交给扩展。这是一种用"约定（事件名 + 返回值协议）"换"灵活（不改源码）"的设计。**

对比旧版的坏例子："这章讲了 28 个事件、注册机制、两阶段绑定"——那是主题列表。

## 3.5 方法论提炼

| 方法名 | 来源 | 复用指引 |
|--------|------|---------|
| **事件总线暴露关键点** | Pi 在 Agent 工作流的 8 个关键点 emit 事件 | 设计插件系统时，识别工作流的关键节点，emit 统一事件。比给每个点写专门 hook 函数扩展性强 |
| **throwing stubs + bindCore 分阶段** | 注册阶段抛错、运行阶段切换为真实实现 | 插件加载时核心还没就绪？用这个模式分离"声明意图"和"执行操作" |
| **API 门面 + 最小权限** | 三层 API（ExtensionAPI / Context / CommandContext） | 给插件不同的门面对象——事件处理时只读，命令处理时可写。永远不暴露内部对象 |
| **错误隔离** | 所有事件处理器都被 try/catch 包裹 | 一个插件崩了不能拖垮核心——错误编码为事件，不打断流程 |

## 4. 认知旅程

### 第一个循环（建→破→立）：扩展能干什么 + 怎么做到

> **阶段 1（建）**：开篇展示 6 个真实需求场景——禁止 rm -rf、用本地 Ollama、SQL 工具、自动翻译、注入时间、/deploy 命令。让读者感到"扩展很强大，能改变 Agent 的方方面面"
>   → 读者感受："哇，扩展能干这么多事，那它到底怎么工作的？"
>
> **阶段 2（破）**：但 Pi 是开源项目，Pi 不可能预先知道你团队要禁 rm -rf。如果 Pi 没有为这个需求写代码，扩展怎么"凭空"加上这个能力？这是一个看似矛盾的问题
>   → 读者感受："对啊，Pi 不知道我的需求，怎么扩展？"
>
> **阶段 3（立）**：答案是**事件总线**——Pi 不需要预先知道你的需求，它只负责在 8 个工作流关键点 emit 事件（"嘿，我要执行工具了，要不要拦？"）。你写扩展回答"要拦"就行。这是一种用"约定"（事件名 + 返回值协议）换"灵活"（不改源码）的设计
>   → 读者感受："原来如此——Pi 不是为每个需求写代码，而是把'决定权'通过事件交给扩展"

### 第二个循环（建→破→立）：扩展怎么写 + 时机矛盾

> **阶段 1（建）**：扩展就是一个 .ts 文件，导出一个工厂函数。在工厂函数里 `pi.on(...)` 注册处理器，看起来很简单
>   → 读者感受："就这？挺简单的"
>
> **阶段 2（破）**：但工厂函数是在 Pi **启动加载阶段**执行的——那时 Agent 还不存在、SessionManager 还没创建、Agent Loop 还没启动。如果你在工厂函数里调 `pi.sendMessage("init")`，会发生什么？消息发给谁？这是一个时机矛盾
>   → 读者感受："对啊，工厂函数这么早执行，运行时的操作怎么办？"
>
> **阶段 3（立）**：Pi 的解法是**两阶段绑定**——工厂函数执行时（注册阶段），`pi` 对象上的操作型方法是 throwing stubs（调用就抛错）；Agent 启动后调用 `bindCore()`，把所有 stubs 替换为真实实现。扩展代码无感知这个差异——你在工厂函数里调会抛错（合理，因为这时调用没意义），在事件处理器里调就是真实操作
>   → 读者感受："妙啊——一个 pi 对象两种状态，自动适配阶段"

### 第三个循环（建→破→立）：扩展崩了怎么办

> **阶段 1（建）**：扩展是第三方代码，写完了就能拦 Agent 行为，能力很大
>   → 读者感受："扩展这么强，那万一写错了呢？"
>
> **阶段 2（破）**：如果扩展的事件处理器抛了异常，会不会拖垮整个 Agent？如果任何扩展都能 forkSession、switchSession、reload，一个恶意扩展会不会劫持 Agent？
>   → 读者感受："对啊，第三方代码不能完全信任"
>
> **阶段 3（立）**：两个机制保护：① **错误隔离**——所有处理器被 try/catch 包裹，扩展崩了只丢弃自己的修改，不影响核心；② **API 门面三层**——ExtensionAPI 给注册能力、Context 给只读查询、CommandContext 给特权操作。普通事件处理器拿不到 forkSession 这种危险方法
>   → 读者感受："考虑得真周到——错误隔离 + 最小权限"

### 收尾
> 总结三大设计（事件总线 / 两阶段绑定 / 错误隔离 + 门面），提炼可迁移的方法论，留悬念给下一章（测试模式）

## 4.5 链路闭环清单

这是 v2 文档最大的漏洞——旧版链路全是断的。新版每个事件必须从触发追踪到最终结果。

| 链路 | 触发端 | 中转 | 消费端 | 最终结果 | 链路闭环？ |
|------|--------|------|--------|---------|----------|
| **rm -rf 拦截** | 用户输入包含 `rm -rf` 的工具调用请求 | Agent Loop 准备调 bash 工具前 → `runner.emit("tool_call")` | 扩展处理器收到 event，识别危险命令，返回 `{block: true, reason}` | Agent Loop 跳过工具执行，返回错误消息给 LLM，LLM 知道命令被拦 | ✅ 必须完整讲 |
| **注入当前时间** | Agent Loop 准备调用 LLM 前 | 构建好 messages → `runner.emitContext(messages)` | 扩展处理器 push 一条 user message，返回新 messages | 新 messages 发给 LLM，LLM 看到时间信息 | ✅ 必须完整讲 |
| **工具结果截断** | 工具执行完产生长结果 | `runner.emitToolResult(event)` | 扩展 A 截断 → 扩展 B 加元数据 → 返回最终 result | Agent Loop 把最终 result 包成 ToolResultMessage 加入 messages | ✅ 用链式传递讲 |
| **会话切换否决** | 用户触发切换会话 | `runner.emit("session_before_switch")` | 扩展弹确认对话框，用户选取消，返回 `{cancel: true}` | Agent 取消切换，留在当前会话 | ✅ 用一票否决讲 |
| **/deploy 命令** | 用户输入 `/deploy` | 斜杠命令路由 → 扩展的 command handler | 处理器拿到 CommandContext，调用 exec 执行部署脚本 | 部署完成，结果通过 sendMessage 回报给 Agent | ✅ 用注册型能力讲 |
